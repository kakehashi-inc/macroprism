# システム仕様

本ドキュメントは MacroPrism のシステム挙動仕様を記載する。

## 名称について

MacroPrism（マクロプリズム）が正式名称・呼称・表示名（productName）である。`mcpm` は略称ではなく、設定ディレクトリ（`~/.mcpm`）・実行ファイル名（`executableName`）・配布物名（`artifactName`）にのみ使用する内部識別子であり、MacroPrism の子音（**M**a**c**ro **P**ris**m**）に対応する。`appId` は `com.macroprism.app`。

管理対象の実体は「プロセス」であり、MCP にもサーバーにも限定されない。`mcp-auth-proxy` はプロセスに関連付け可能な任意の認証機能（外部ツール）である。ディープリンク用プロトコルは `macroprism://`。

## 設定ファイルのレイアウト移行

`~/.mcpm/config.json` のプロセス定義キーは `processes` である。`ConfigManager.loadConfig()` は読み込み時に
`processes` キーが無く旧レイアウトのキー `mcpServers` が存在する場合、その内容を `processes` として取り込み、
直ちに新レイアウトで保存し直す（自動移行）。新レイアウトの config.json を旧バージョンで読むこと
（ダウングレード）は非対応。

## ビルド成果物 (Windows)

`yarn dist:win` / `yarn release:win` 実行時の Windows 向け成果物:

| ターゲット | 成果物 | メタデータ |
|---|---|---|
| NSIS | `mcpm-${version}-win-${arch}.exe`, `mcpm-${version}-win-${arch}.exe.blockmap` | `latest.yml` |
| portable | `mcpm-${version}-win-${arch}-portable.zip` | なし |

### portable 版の zip 配布

`electron-builder` の `afterAllArtifactBuild` フック (`scripts/zip-portable.js`) によって、
ビルド完了後に portable 版 `.exe` を `.zip` に圧縮し、元の `.exe` を削除する。

判定ロジック:

- `BuildResult.platformToTargets` から Windows の portable ターゲット存在を確認
- `artifactPaths` から `.exe` を抽出し、同 basename の `.blockmap` が **存在しない** ものを portable 版と確定
  (NSIS は `.exe` と `.exe.blockmap` のペアを必ず出力するため、blockmap が無い `.exe` は portable 版で確定)
- `archiver` で zip 圧縮後、元の `.exe` を削除
- 生成 zip パスを publish 対象に追加して返却

### zip 配布する理由

未署名の `.exe` を直接配布すると、ブラウザのダウンロード警告および Windows Defender SmartScreen /
ウイルス対策ソフトによるブロックが発生しやすい。zip に格納することで配布経路上の警告を緩和する。

### 他プラットフォーム

macOS / Linux ビルド時は `BuildResult.platformToTargets` に Windows portable ターゲットが含まれないため、
`afterAllArtifactBuild` フックは何もせず素通しする (NSIS 版の `.exe` / `.blockmap` / `latest.yml` 等も
一切変更しない)。

## HTTPS プロキシ

### 設定モデル (プロキシ名 / 複数ホスト名 / 複数ポート転送)

HTTPS プロキシは **プロキシ名** をキーに定義する (旧仕様の「ホスト名をキー」から変更)。1 プロキシは以下を持つ。

- `bindMode: 'local' | 'all' | 'custom'` — バインドアドレスの指定方式 (既定 `local`)。
  - `local`: ローカルのみ。`127.0.0.1` と `::1` で待ち受ける (`HTTPS_PROXY_LOCAL_BIND_ADDRESSES`)。
  - `all`: 全アドレス。`0.0.0.0` (IPv4 全アドレス) と `::` (IPv6 全アドレス) で待ち受ける
    (`HTTPS_PROXY_ALL_BIND_ADDRESSES`)。`::` は IPv6 専用 (`ipv6Only: true`) でバインドし、
    デュアルスタックによる `0.0.0.0` との同一ポート衝突を避ける (custom で `::` を指定した場合も同様)。
  - `custom`: `bindAddresses: string[]` に列挙した任意のアドレス。個数制限なし。UI で追加・編集・
    削除でき、1 件以上が必須。重複は起動時に除去する。
  UI の一覧では `local`/`all` はタイプ名を、`custom` は設定値そのものを表示する。
- `hostnames: string[]` — 証明書の SAN 兼、URL 書き換えのホスト一致判定に使うホスト名群。
  ワイルドカード (例 `*.example.local`) を含められる。
- `portMappings: { from, to }[]` — ポート転送定義 (複数可)。`from` は転送先 HTTP ポート、
  `to` は HTTPS 待ち受けポート。**マッピング × バインドアドレス** ごとに独立したサーバーが
  `to` で待ち受け、`http://127.0.0.1:<from>` へ転送する (`HttpsProxyManager.servers` は
  名前→サーバー配列。TLS 終端/リダイレクト用の内部サーバーはマッピング単位で共有)。
- `autoStart?: boolean`。

### ポート待ち受けと HTTP→HTTPS リダイレクト

待ち受けは素の `net.Server` で行い、接続の**先頭 1 バイト**で振り分ける (`0x16` = TLS ハンドシェイク)。

- TLS → TLS 終端する `https.Server` (express + express-http-proxy) へ渡し、バックエンドへ転送する。
- 平文 HTTP → 待ち受けなしの `http.Server` へ渡し、**バックエンドへは転送せず** 同一 URL の
  `https://` へ `301` リダイレクトを返す (Host ヘッダのホスト:ポートとパス/クエリを維持)。
  `http://localhost:8443/...` のようにブラウザが HTTPS ポートへ http でアクセスした場合の救済。

データが届かないまま放置されたソケットは 10 秒でタイムアウト破棄する (振り分け後は各サーバー管理)。
停止時は内部サーバーの キープアライブ接続を `closeAllConnections` で破棄してから待ち受けを閉じる。

これにより、同一 `localhost` 上の複数ポートを 1 プロキシで扱え、複数ホスト名やワイルドカード証明書を
事前に用意できる。ホスト一致は宣言済みホスト名で判定するため **DNS/IP 解決は行わない**。

### 証明書 (ローカルCA方式)

証明書は node-forge で発行する。**ローカル CA** を 1 つ生成し、全プロキシのサーバー証明書を
その CA の秘密鍵で署名する 2 階層構成。

- **CA**: `~/.mcpm/certs/ca/{ca-cert.pem,ca-key.pem}`。CN `MacroPrism Local CA`、RSA 2048、
  有効期間 10 年、`basicConstraints CA:true` / `keyUsage keyCertSign,cRLSign` (いずれも critical)。
  無い、または残り有効期間が 30 日未満のときに `ensureCa` が生成し直す。秘密鍵は mode 0o600 で
  書き込む (Windows では ACL 依存)。CA 秘密鍵はエクスポート対象にしない。
- **サーバー証明書**: プロキシ名ごとに `~/.mcpm/certs/<プロキシ名>/{cert.pem,key.pem}`
  (名前はファイル名向けにサニタイズ)。SAN には全 `hostnames` を DNS 名として (ワイルドカードも
  そのまま) 登録し、ループバック IP (127.0.0.1 / ::1) も付与する。CN は先頭ホスト名。RSA 2048、
  既定 90 日、`extKeyUsage serverAuth`、CA 署名 (sha256)。

起動時、サーバー証明書が「無い / 期限切れ / 宣言ホスト名と SAN が不一致 / **現在の CA の署名でない**」
のいずれかであれば自動再発行する (`X509Certificate.verify` で CA 公開鍵に対する署名を検証)。
これにより旧方式 (自己署名) の証明書や、CA 再生成前の証明書は次回起動時に自動で切り替わる。
ホスト名の追加・削除は SAN 集合の完全一致判定により次回起動時に反映され、UI の「証明書を再作成」
でも即時反映できる。

画面上部の「CA証明書をダウンロード」ボタンで CA 証明書を保存できる (IPC
`https-proxy:export-ca-cert` → `HttpsProxyManager.exportCaCertificate` が保存ダイアログの指定先へ
コピー。既定ファイル名 `macroprism-ca.crt`)。OS の信頼ストア (信頼されたルート証明機関) への登録は
**ユーザー自身が行う** (アプリは登録しない)。一度登録すれば、プロキシの追加や証明書の再発行があっても
CA が同じ限り再登録は不要で、ブラウザ警告が消え Service Worker など完全に信頼された HTTPS を要求する
機能も使える。

### URL 書き換え

バックエンド (HTTP) が返す絶対 URL を、ブラウザが HTTPS を維持できるよう書き換える。リダイレクト
(Location ヘッダ) とレスポンス本文で **同一ルール** を適用する。本文の書き換え対象は
Content-Type が `text/html` / `json` を含むもの (`application/json` 等) / `text/plain` の
レスポンス (XHR/fetch の応答も同じ経路を通るため区別なく対象)。実装は
`HttpsUrlRewriter` に集約し、`HttpsProxyManager` の `userResHeaderDecorator` (Location) と
`userResDecorator` (本文) から呼び出す。

対象は **スキーマが `http`** の絶対 URL のみ (`https` はそのまま)。以下を **すべて満たす** 場合に限り、
ホスト名を維持したまま スキーマを `https`、ポートを対応する `to` に差し替える。

- **ホスト一致**: URL のホストが宣言済み `hostnames` に一致する (完全一致、または `*.suffix` の
  ワイルドカード一致=1 ラベルのみ)。→ 事前宣言したホストのみ対象。外部サーバーは対象外。
- **ポート一致**: URL のポートが `portMappings` の `from` または `to` に一致する
  (別ポートの別サービスは対象外)。ポート省略時は 80 とみなす。
  - `from` に一致 → 対応する `to` へ。`to` に一致 → その `to` (スキーマのみ変更)。

出力ポートが 443 のときはポートを省略する (`https://host`)。IPv6 リテラルはブラケット表記
(`https://[::1]:8443`) を維持する。本文置換は `http://host[:port]` オリジンをユニーク単位で
判定・置換する (パス以降は変更しない)。

圧縮の扱い: `gzip` は `express-http-proxy` が自動展開/再圧縮するため書き換え対象。gzip/無圧縮
以外の Content-Encoding (br/zstd 等) はホワイトリスト方式で素通しし (壊れた本文を返さない)、
スキップした旨をログに記録する。

### 設定の自動移行

旧レイアウト `{ [hostname]: { forwardPort, listenPort, autoStart } }` は、起動時に
`migrateHttpsProxies` が新レイアウトへ変換する (`ConfigManager.loadConfig`)。旧キー (ホスト名) を
そのままプロキシ名として引き継ぎ、`hostnames: [旧hostname]`、`portMappings: [{ from: forwardPort,
to: listenPort }]` を生成する。また `bindMode` を持たない既存データは、`bindAddresses` が
未設定またはローカル既定値 (`127.0.0.1` / `::1`) と一致すれば `local`、それ以外は `custom` として
引き継ぐ。変換が発生した場合は即座に保存し、ユーザーの操作なしで新機能へ移行する。
