# システム仕様

本ドキュメントは MacroPrism のシステム挙動仕様を記載する。

## 名称について

MacroPrism（マクロプリズム）が正式名称・呼称・表示名（productName）である。`mcpm` は略称ではなく、設定ディレクトリ（`~/.mcpm`）・実行ファイル名（`executableName`）・配布物名（`artifactName`）にのみ使用する内部識別子であり、MacroPrism の子音（**M**a**c**ro **P**ris**m**）に対応する。`appId` は `com.macroprism.app`。

管理対象の実体は「プロセス」であり、MCP にもサーバーにも限定されない。`mcp-auth-proxy` はプロセスに関連付け可能な任意の認証機能（外部ツール）である。ディープリンク用プロトコルは `macroprism://`。

## 設定ファイルのレイアウト移行

`~/.mcpm/config.json` のプロセス定義キーは `processes` である。`ConfigManager.loadConfig()` は読み込み時に
`processes` キーが無く旧レイアウトのキー `mcpServers` が存在する場合、その内容を `processes` として取り込み、
直ちに新レイアウトで保存し直す（自動移行）。新レイアウトの config.json を旧バージョンで読むこと
（ダウングレード）は非対応。

## ビルド成果物 (Windows)

`yarn dist:win` / `yarn release:win` 実行時の Windows 向け成果物:

| ターゲット | 成果物 | メタデータ |
|---|---|---|
| NSIS | `mcpm-${version}-win-${arch}.exe`, `mcpm-${version}-win-${arch}.exe.blockmap` | `latest.yml` |
| portable | `mcpm-${version}-win-${arch}-portable.zip` | なし |

### portable 版の zip 配布

`electron-builder` の `afterAllArtifactBuild` フック (`scripts/zip-portable.js`) によって、
ビルド完了後に portable 版 `.exe` を `.zip` に圧縮し、元の `.exe` を削除する。

判定ロジック:

- `BuildResult.platformToTargets` から Windows の portable ターゲット存在を確認
- `artifactPaths` から `.exe` を抽出し、同 basename の `.blockmap` が **存在しない** ものを portable 版と確定
  (NSIS は `.exe` と `.exe.blockmap` のペアを必ず出力するため、blockmap が無い `.exe` は portable 版で確定)
- `archiver` で zip 圧縮後、元の `.exe` を削除
- 生成 zip パスを publish 対象に追加して返却

### zip 配布する理由

未署名の `.exe` を直接配布すると、ブラウザのダウンロード警告および Windows Defender SmartScreen /
ウイルス対策ソフトによるブロックが発生しやすい。zip に格納することで配布経路上の警告を緩和する。

### 他プラットフォーム

macOS / Linux ビルド時は `BuildResult.platformToTargets` に Windows portable ターゲットが含まれないため、
`afterAllArtifactBuild` フックは何もせず素通しする (NSIS 版の `.exe` / `.blockmap` / `latest.yml` 等も
一切変更しない)。

## HTTPS プロキシ

### 設定モデル (プロキシ名 / 複数ホスト名 / 複数ポート転送)

HTTPS プロキシは **プロキシ名** をキーに定義する (旧仕様の「ホスト名をキー」から変更)。1 プロキシは以下を持つ。

- `hostnames: string[]` — 証明書の SAN 兼、URL 書き換えのホスト一致判定に使うホスト名群。
  ワイルドカード (例 `*.example.local`) を含められる。
- `portMappings: { from, to }[]` — ポート転送定義 (複数可)。`from` は転送先 HTTP ポート、
  `to` は HTTPS 待ち受けポート。各マッピングごとに独立した HTTPS サーバーが `to` で待ち受け、
  `http://127.0.0.1:<from>` へ転送する (`HttpsProxyManager.servers` は名前→サーバー配列)。
- `autoStart?: boolean`。

これにより、同一 `localhost` 上の複数ポートを 1 プロキシで扱え、複数ホスト名やワイルドカード証明書を
事前に用意できる。ホスト一致は宣言済みホスト名で判定するため **DNS/IP 解決は行わない**。

### 証明書

プロキシ名ごとに `~/.mcpm/certs/<プロキシ名>/{cert.pem,key.pem}` を保持する
(名前はファイル名向けにサニタイズ)。自己署名証明書の SAN には全 `hostnames` を DNS 名として
(ワイルドカードもそのまま) 登録し、ループバック IP (127.0.0.1 / ::1) も付与する。CN は先頭ホスト名。
起動時、証明書が「無い / 期限切れ / 宣言ホスト名と一致しない」いずれかであれば自動再生成する。
一致判定は証明書の DNS SAN 集合と宣言ホスト名集合の完全一致で行うため、ホスト名の**追加・削除**
どちらでも次回起動時に証明書へ反映される (削除したホスト名は SAN から除去される)。稼働中プロキシを
更新した場合は再起動時に反映され、UI の「証明書を再作成」でも即時反映できる。

### URL 書き換え

バックエンド (HTTP) が返す絶対 URL を、ブラウザが HTTPS を維持できるよう書き換える。リダイレクト
(Location ヘッダ) とレスポンス本文 (HTML/JS/JSON) で **同一ルール** を適用する。実装は
`HttpsUrlRewriter` に集約し、`HttpsProxyManager` の `userResHeaderDecorator` (Location) と
`userResDecorator` (本文) から呼び出す。

対象は **スキーマが `http`** の絶対 URL のみ (`https` はそのまま)。以下を **すべて満たす** 場合に限り、
ホスト名を維持したまま スキーマを `https`、ポートを対応する `to` に差し替える。

- **ホスト一致**: URL のホストが宣言済み `hostnames` に一致する (完全一致、または `*.suffix` の
  ワイルドカード一致=1 ラベルのみ)。→ 事前宣言したホストのみ対象。外部サーバーは対象外。
- **ポート一致**: URL のポートが `portMappings` の `from` または `to` に一致する
  (別ポートの別サービスは対象外)。ポート省略時は 80 とみなす。
  - `from` に一致 → 対応する `to` へ。`to` に一致 → その `to` (スキーマのみ変更)。

出力ポートが 443 のときはポートを省略する (`https://host`)。IPv6 リテラルはブラケット表記
(`https://[::1]:8443`) を維持する。本文置換は `http://host[:port]` オリジンをユニーク単位で
判定・置換する (パス以降は変更しない)。`gzip` は `express-http-proxy` が自動展開/再圧縮するため
書き換え対象。`br` / `deflate` は非対応のため素通しする。

### 設定の自動移行

旧レイアウト `{ [hostname]: { forwardPort, listenPort, autoStart } }` は、起動時に
`migrateHttpsProxies` が新レイアウトへ変換する (`ConfigManager.loadConfig`)。旧キー (ホスト名) を
そのままプロキシ名として引き継ぎ、`hostnames: [旧hostname]`、`portMappings: [{ from: forwardPort,
to: listenPort }]` を生成する。変換が発生した場合は即座に保存し、ユーザーの操作なしで新機能へ移行する。
